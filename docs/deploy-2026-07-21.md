# Deploy notes — 2026-07-21 (employee module + RBAC phase 3)

This release ships the employee timesheet home, working punch in/out with GPS,
the employee help chat, and RBAC phase 3 (PII encryption at rest + department
write scoping). It has **environment and database prerequisites** — read before
deploying.

## 1. Required environment variables (Vercel → Production)

| Variable | Value | Why |
|---|---|---|
| `FIELD_ENCRYPTION_KEY` | a 32+ char secret (e.g. a 64-hex string) | **Required in production** — the build fails without it (fail-closed PII encryption, SEC-008). |
| `DATABASE_URL` | the Supabase **pooler** URL (`...pooler.supabase.com:6543`) | The direct host (`db.<ref>.supabase.co`) is IPv6-only and fails on many networks. |
| `GEMINI_API_KEY` | your Gemini key | Employee help chat. |
| `LLM_PROVIDER` | `gemini` | Selects Gemini as the chat model. |
| `PLATFORM_ADMIN_EMAILS` | comma-separated operator emails | Gates `/super-admin` (SEC-002). |
| `DEMO_MODE` | `true` **only** for a demo/prospect environment | Enables the fixed Taazur demo logins. **Leave unset/false for a real customer tenant** — the passwords are public. |

The build was verified to compile for production with `FIELD_ENCRYPTION_KEY` set.
If the Vercel build fails with a `FIELD_ENCRYPTION_KEY` env error, add the
variable and redeploy — the previous deployment stays live until then.

## 2. Database migration (per environment / per Supabase project)

The attendance tables use a per-tenant schema managed by a script, not by
`drizzle-kit migrate`. This release adds `punch_sequence`, `punch_in_lat`,
`punch_in_lng`, `punch_in_accuracy`, and reconciles legacy column names
(`status` → `attendance_status`, `exception_type` → `attendance_exception_type`)
and column defaults.

Run once against each Supabase project the app uses:

```
node --env-file=.env scripts/complete-db-attendance.mjs
```

**Status:** already run against the shared Supabase project this app currently
uses, so the live database is up to date. Re-run it for any new environment or
after provisioning new tenant schemas. (Idempotent — safe to re-run.)

> Known gap (pre-existing, not fixed here): `generateTenantDDL()` in
> `packages/db/src/tenant-manager.ts` only creates `departments` + `employees`,
> so brand-new tenants get no `attendance_records` table at all. Provisioning
> new tenants needs a fuller DDL step — tracked separately.

## 3. Follow-up: encrypt existing PII (SEC-008)

New writes to iqama / passport / IBAN are encrypted automatically. Existing rows
stay readable as legacy plaintext until backfilled. To encrypt them at rest, run
**in an environment that has the production `FIELD_ENCRYPTION_KEY`** (never with
the dev fallback key, or the data becomes unreadable in production):

```
FIELD_ENCRYPTION_KEY=<prod key> node --env-file=.env scripts/encrypt-pii-backfill.ts
```

Until the backfill runs, the iqama duplicate check compares encrypted input
against plaintext rows, so a duplicate national ID could slip through — run the
backfill promptly after the key is set.

> Do not rotate `FIELD_ENCRYPTION_KEY` after data is encrypted without a
> re-encryption plan — a key mismatch makes encrypted PII reads throw.
